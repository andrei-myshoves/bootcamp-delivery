# Скриншотные (визуальные) тесты

Визуальное регрессионное тестирование построено на **Storybook + Vitest Browser Mode (Playwright)**.

Каждая story автоматически превращается в тест (`@storybook/addon-vitest` → `storybookTest`), рендерится в реальном Chromium, и с неё снимается скриншот в нескольких разрешениях. Снимок сравнивается с эталоном; при расхождении тест падает и сохраняется diff.

> Эталоны генерируются и сравниваются **только в Linux-окружении (Docker)**, чтобы локальный прогон и CI давали идентичный рендеринг. Поэтому дев-команды визуальных тестов запускаются через Docker.

---

## Требования: Docker через Colima

Для локального прогона визуальных тестов нужен запущенный Docker. На macOS используется **Colima** как рантайм Docker.

Установка (один раз):

```bash
brew install colima docker
```

**Перед запуском визуальных тестов обязательно поднимите Colima:**

```bash
colima start
```

Проверить, что Docker доступен:

```bash
docker ps
```

Если `docker ps` ругается на отсутствие демона — значит Colima не запущена, выполните `colima start`. Остановить рантайм при необходимости: `colima stop`.

## Команды

| Команда                      | Что делает                                          | Где выполняется    |
| ---------------------------- | --------------------------------------------------- | ------------------ |
| `npm run test:visual`        | Прогон визуальных тестов (сверка с эталонами)       | **Docker (Linux)** |
| `npm run test:visual:update` | Обновить/создать эталоны                            | **Docker (Linux)** |
| `npm run report`             | Открыть HTML-отчёт последнего прогона               | локально           |
| `npm run ci:test:visual`     | Нативный прогон (используется в CI и внутри Docker) | нативно            |

> `ci:test:visual` напрямую руками обычно не запускают — это «движок», который вызывают CI и Docker-обёртки. Дев-команды без префикса (`test:visual`, `test:visual:update`) всегда идут через Docker.

## Флоу разработки

1. Поднимите Docker-рантайм: `colima start`.
2. Пишете/меняете компонент и его story (`*.stories.tsx`).
3. Прогоняете визуальные тесты:

    ```bash
    npm run test:visual
    ```

4. Если тест **упал**:
    - откройте отчёт `npm run report` (или посмотрите картинки в `.storybook/screenshots/.diffs/`) и сравните `reference` / `actual` / `diff`;
    - **изменение ожидаемое** (вы намеренно поменяли вёрстку или добавили story) → обновите эталоны:

        ```bash
        npm run test:visual:update
        ```

        затем закоммитьте обновлённые PNG из `.storybook/screenshots/references/`;

    - **изменение случайное** (регресс) → правьте код, пока тест не станет зелёным.

5. Коммитите код вместе с актуальными эталонами.

> Первый прогон новой story создаёт эталоны и падает (эталона ещё нет) — это нормально. Достаточно один раз выполнить `npm run test:visual:update`, который создаёт **все** недостающие эталоны за один проход.

## Где лежат скриншоты

```
.storybook/screenshots/
├── references/        # эталоны — КОММИТЯТСЯ
│   └── <story-файл>/<Story>-<viewport>-chromium-linux.png
├── .actual/           # снимки последнего прогона (в .gitignore)
├── .diffs/            # diff + actual при падениях (в .gitignore)
└── report/            # HTML-отчёт (в .gitignore)
```

- Коммитим только `references/`.
- `.actual/`, `.diffs/`, `report/` — артефакты прогона, в `.gitignore`.
- Разрешения для съёмки задаются в `.storybook/vitest.setup.ts` (сейчас `desktop` и `mobile`).

---

## CI

В Pull Request визуальные тесты прогоняются нативно командой `ci:test:visual` (раннер CI и так Linux). При падении diff-картинки и HTML-отчёт выгружаются в артефакты PR (`visual-report`) — их можно скачать и проанализировать.

Поскольку эталоны хранятся как `*-chromium-linux.png`, важно обновлять их через `npm run test:visual:update` (Docker = тот же Linux), иначе CI не найдёт нужный эталон.
